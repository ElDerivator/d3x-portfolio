# D3X Public Intake Security Model

## Decision

`d3x.biz` is a public intake surface only. It must never send lead, prospect, customer, request, pricing, support, or operational data directly to `d3x-orch`, `d3x-data`, `d3x-compute`, or any internal D3X service.

Public intake is allowed only when the public surface submits into a safe reception layer such as Google Workspace through Google Apps Script or another controlled public intake endpoint. Internal D3X systems may later absorb from Workspace through governed importers.

## Forbidden Direct Ingestion

The following flow is explicitly blocked:

```text
d3x.biz public form
-> d3x-orch direct API call
```

Direct `d3x.biz -> d3x-orch` ingestion is forbidden because it would expose the internal control plane to untrusted internet input, collapse the public/internal boundary, make lead data immediately operational before policy review, increase attack surface, and create pressure to expose internal URLs, private hostnames, LAN addresses, Tailscale endpoints, credentials, or proxy routes.

## Approved Intake Path

The approved flow is:

```text
d3x.biz public form
-> Google Apps Script or controlled public intake endpoint
-> Google Sheets / Google Workspace
-> internal D3X Workspace Importer
-> Policy Gate
-> L0/L1/L2 memory
-> Agency Runtime
-> draft-only commercial tasks
-> human approval
-> outbound response/cotizacion/follow-up
```

## Public Boundary

The public boundary includes:

- `d3x.biz`
- Squarespace pages and custom code
- public form HTML
- public JavaScript
- public CSS
- uploaded public images and static assets
- any public intake endpoint explicitly designed for internet input

The public boundary may collect limited lead information and submit it only to the approved reception layer.

## Internal D3X Boundary

The internal D3X boundary includes:

- `d3x-orch`
- `d3x-data`
- `d3x-compute`
- LAN addresses
- Tailscale addresses
- localhost services
- private hostnames
- internal APIs
- policy gates
- memory layers
- agency runtime
- internal importers

No public page, form, script, image, static asset, or public proxy may reveal or call this boundary directly.

## Allowed Public Data Collection Fields

The public intake surface may collect only fields needed for initial commercial triage:

- name
- company or project name
- email
- phone, optional
- preferred contact method
- service interest
- short project description
- approximate budget range, optional
- timeline or urgency, optional
- consent checkbox
- source page or campaign tag
- timestamp generated by the intake layer

## Disallowed Public Data And Secrets

The public intake surface must not request, expose, store in client-side code, or transmit directly to D3X:

- API keys
- tokens
- passwords
- private URLs
- LAN IPs
- Tailscale URLs or IPs
- localhost URLs
- private hostnames
- internal task IDs
- internal agent IDs
- internal memory IDs
- raw credentials
- payment card data
- government IDs
- sensitive personal documents
- production system access details
- files that contain secrets

## Hard Constraints

- Public website must not call `d3x-orch`.
- Public website must not call `d3x-data`.
- Public website must not call `d3x-compute`.
- Public website must not expose LAN IPs.
- Public website must not expose Tailscale URLs.
- Public website must not expose localhost URLs.
- Public website must not expose private hostnames.
- Public website must not expose API keys or tokens.
- Public website must not create direct public proxy routes into D3X.
- Public website must not send emails automatically.
- D3X must not contact clients without human approval.

## Workspace As Buffer Zone

Google Workspace acts as the reception and buffer zone between untrusted public input and internal D3X execution.

Workspace provides a reviewable, auditable, rate-limited, human-visible staging area. It allows incoming public leads to be captured without exposing D3X infrastructure, without granting the public site access to internal services, and without turning every form submission into an operational task.

## D3X Absorption

D3X may absorb public intake only from Workspace through an internal Workspace Importer. The importer runs inside the internal boundary and must apply:

- source validation
- schema validation
- consent checks
- deduplication
- classification
- risk flags
- policy gate evaluation
- controlled memory writes
- draft-only task creation

## Human Approval

All commercial actions created from public intake must remain draft-only until approved by a human.

D3X must not automatically contact clients, send quotes, send follow-ups, start negotiations, make promises, or trigger outbound communication without explicit human approval.

## v0.1 Limitations

In v0.1:

- `d3x.biz` is an intake and presentation surface only.
- Google Workspace is the only approved reception buffer.
- D3X has no direct public ingestion route.
- No automatic email replies are allowed.
- No direct lead-to-runtime automation is allowed.
- Internal importers may create only draft commercial tasks.

## v0.2 Future Path

In v0.2:

- Add a governed Workspace Importer with explicit schemas.
- Add duplicate detection for repeated leads.
- Add lead status tracking in Workspace.
- Add policy labels for commercial risk.
- Add review queues for human approval.
- Add structured draft quote generation with no outbound sending.

## v0.3 Future Path

In v0.3:

- Add a controlled public intake endpoint only if it remains outside internal D3X services.
- Add stronger abuse controls, rate limits, captcha, and origin checks.
- Add signed handoff from intake endpoint to Workspace.
- Add importer observability and audit trails.
- Add human-approved outbound workflows.
- Keep `d3x-orch`, `d3x-data`, and `d3x-compute` private.

## Enforcement Notes

Any public file containing internal URLs, LAN addresses, Tailscale addresses, localhost URLs, private hostnames, API keys, tokens, or direct calls into D3X is a blocker for publication.

Development-only tools such as local servers must not be deployed as part of the public website and must not be referenced by public-facing site content.
